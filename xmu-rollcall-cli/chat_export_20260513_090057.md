### 二、核心发现

**发现 1：QR 编码的是 URL，不是 token 字符串**

HTML 源码中最关键的一行：

```
<qrcode class="QRCode" size="110" version="6" 
        data="https://lnt.xmu.edu.cn:443/mob...">
```

QR 码的内容是一个指向 `lnt.xmu.edu.cn:443/mob...` 的 URL。这是"扫码跳转"模式——学生手机扫码后打开网页，由前端 JS 解析 URL 参数，再调 answer API。这与数字签到的"直接从 API 提取 numberCode"完全不同。

**发现 2：student_rollcalls 中没有 QR token**

Strategy 10 对 student_rollcalls 做了深度扫描（331 个匹配），但没有报出任何成功的 answer。所有 331 个匹配都是 `department.code`、`klass.code`、`status`、`rollcall_status` 等组织/状态字段，**没有** `qr_code`、`qr_token` 这类可直接提交的凭证字段。

这意味着：QR 签到的 token **不在** student_rollcalls API 里。它只存在于 QR 码编码的 URL 中。

**发现 3：WebSocket (Socket.IO) 可达**

```
WS upgrade /socket.io/ → 101 (Switching Protocols)
```

Socket.IO 握手成功。token 极可能是老师端通过 WebSocket 实时推送的——老师点击"开始签到"→ 服务端生成 token → 通过 WS 推到前端 → 前端渲染 QR 码。这对逆向的影响是：token 不会出现在 REST API 响应里，但一定会出现在前端 JS 上下文或 WebSocket 消息中。

**发现 4：JS Bundle 中有 QR 签到逻辑**

78 个 JS 文件中，3.2MB 的 `55054-b3dcb926.js`（i18n 文件）包含：

```
rollcallType: {"merged":"合併生成","newCapec":"新開普","manual":"手動點名","selfRegistration":"自助註冊"}
```

更大、更关键的业务逻辑在 `30803-7b1f840f.js`（1.05MB）、`61409-cf835457.js`（786KB）、`68924-a30f5597.js`（283KB）中。这些文件里有 5 处 `/api/rollcall/` 引用——QR 签到的完整调用链就在里面。

### 三、五条逆向路线评估

| 路线 | 可行性 | 难度 | 说明 |
|---|---|---|---|
| **A. QR URL 截获** | ✅ 高 | 低 | 直接在 HTML 中提取 QR 标签的 `data` 属性即可获得完整 URL，无需扫码 |
| **B. WebSocket 监听** | ✅ 高 | 中 | 连接 `/socket.io/`，监听 rollcall 相关事件，在 token 推送时截获 |
| **C. JS 逆向** | ✅ 高 | 中-高 | 下载业务 JS bundle，格式化后搜索 `answer`/`qr`/`rollcall`，提取 payload 结构和端点 |
| **D. 移动端抓包** | ✅ 中 | 高 | 用 mitmproxy 截获 TronClass App 的 HTTPS 流量，直接观察完整请求 |
| **E. QR 图像解码** | ❌ 低 | — | QR 是前端 `<qrcode>` 组件动态渲染的，无静态图片端点（Strategy 15 未命中） |

### 四、推荐实施路径（按优先级）

**第一步：截获完整 QR URL（零成本）**

修改 Strategy 3，不截断 URL，完整输出 `<qrcode data="...">` 的值。这个 URL 大概率是：

```
https://lnt.xmu.edu.cn:443/mobile/rollcall/{id}?token=xxx
```

或类似格式。拿到完整 URL 后，解析参数即可获得 token。

**第二步：用截获的 token 尝试 answer**

payload 格式最可能是以下之一（参考雷达签到的 `deviceId` 模式）：

```json
{"deviceId": "<uuid>", "qrToken": "<token>"}
{"deviceId": "<uuid>", "qrCode": "<token>"}
{"deviceId": "<uuid>", "token": "<token>"}
{"deviceId": "<uuid>", "code": "<token>"}
```

目标端点：`PUT /api/rollcall/{id}/answer`（CORS 已确认 PUT 可用）

**第三步（备选）：WebSocket 监听**

如果 URL 中不含 token（只是 `mobile/rollcall/{id}`），那 token 是通过 WebSocket 实时推送的。用 Python 的 `python-socketio` 连接 `/socket.io/`，监听 `rollcall_token` / `qr_update` / `token` 等事件名。

**第四步（备选）：JS Bundle 逆向**

下载 `30803-7b1f840f.js`，用 prettier 格式化，搜索：
- `PUT.*answer`
- `deviceId`
- `qrCode` / `qrToken`
- `answerQrRollcall`

找到完整调用后，直接在 Python 端复现。

### 五、结论

**二维码签到可以逆向，而且难度低于预期。**

核心原因是这个系统把 QR 码渲染在了前端 HTML 中——`<qrcode data="URL">` 的 `data` 属性直接暴露了 token 所在的 URL。与数字签到不同（token 嵌在 student_rollcalls 的深层 JSON 里），QR 的 token 出现在页面的 HTML 源码里，实际上更容易提取。

最关键的下一步就是**完整输出那个被截断的 URL**——它出现在首页 HTML 第 958425 字节附近，以 `https://lnt.xmu.edu.cn:443/mob` 开头。拿到完整 URL 后，QR 签到逆向基本就完成了 80%。
